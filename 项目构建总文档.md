# AI狼人杀（Vue3全生态）前端项目构建总文档

## 1. 文档目的

本文档用于指导 AI狼人杀 纯文字网页版前端的完整构建、模块拆分、接口约定、开发顺序与交付标准。它的目标不是只描述功能，而是把前端项目如何从零搭建到可上线的过程讲清楚，方便开发人员直接照着实现。

## 2. 项目定位

### 2.1 产品定位

当前版本只做纯文字狼人杀网页端，不包含语音、不包含原生客户端、不包含真人联机。核心目标是先做出一套稳定的单局对战体验，让 1 名真人玩家与 4 到 5 名 AI 完成完整对局。

### 2.2 前端职责边界

前端只负责以下内容：

- 页面展示
- 用户交互
- 数据渲染
- 本地状态管理
- 与后端的接口通信

前端不负责以下内容：

- 身份分配规则
- 夜间行动裁决
- AI 发言生成
- 投票结果计算
- 胜负判定

这些逻辑全部由后端统一输出，前端仅消费结果。

### 2.3 设计原则

- 前后端彻底分离
- 模块化和可扩展优先
- 所有行为围绕对局流程组织
- 先定义类型和接口，再实现页面
- 结构预留给后续语音、多端和联机扩展

## 3. 技术方案

### 3.1 固定技术栈

| 模块 | 选型 | 作用 |
|---|---|---|
| 框架 | Vue 3 + Vite | 页面渲染、组件化开发、快速构建 |
| 语言 | TypeScript | 类型约束、提高可维护性 |
| 状态管理 | Pinia | 管理对局状态、玩家信息、聊天记录 |
| 路由 | Vue Router | 管理首页、房间页、对局页、结算页 |
| UI 库 | Element Plus | 快速搭建面板、按钮、弹窗、表单 |
| 通信 | Axios + WebSocket | REST 接口与实时消息通道 |
| 规范 | ESLint + Prettier | 统一编码风格 |

### 3.2 环境变量

建议在 `.env.development` 与 `.env.production` 中统一维护以下配置：

- `VITE_API_BASE_URL`
- `VITE_WS_URL`
- `VITE_APP_TITLE`
- `VITE_ENABLE_MOCK`

### 3.3 建议脚本

- `dev`：本地开发
- `build`：生产构建
- `preview`：预览构建结果
- `lint`：代码检查

## 4. 整体架构

### 4.1 分层模型

```plain text
View      -> 页面与组件，只关心展示和交互
Store     -> Pinia，全局状态与业务数据
Comms     -> Axios / WebSocket，对接后端
Utils     -> 格式化、校验、常量、辅助方法
Router    -> 路由与访问控制
Types     -> 全局类型定义
```

### 4.2 数据流

1. 用户在页面上进行操作。
2. 页面调用组件或 Store 的 action。
3. action 通过 REST 或 WebSocket 请求后端。
4. 后端返回状态、事件或结果。
5. Store 更新后驱动页面重新渲染。

这个数据流必须保持单向，避免多个组件各自维护一份状态。

## 6. 页面与组件规划

### 6.1 页面

- 首页：创建游戏、加入游戏、查看基础说明
- 房间页：等待开局、展示房间成员、展示准备状态
- 对局页：对局主流程，包含发言、投票、身份展示和状态提示
- 结算页：展示胜负结果、角色明细、复盘入口

### 6.2 核心组件

- ChatBox：承载文字发言与输入框
- VotePanel：负责投票候选展示与选择
- RoleCard：展示自己的身份
- Announce：展示系统公告、阶段切换和结局提示
- PlayerList：展示所有存活玩家与死亡状态
- GameStatus：展示当前对局阶段与当前轮次

## 7. 类型系统设计

建议建立统一的 `types/game.ts`，作为所有接口、Store 和组件的数据基础。

### 7.1 核心类型

- GameStatus：waiting、night、day、speak、vote、end
- RoleType：wolf、civilian、prophet、unknown
- Player：玩家基础信息
- ChatMessage：发言记录
- VoteData：投票记录
- SocketMessage：WebSocket 消息统一结构

### 7.2 设计原则

- 类型必须可复用
- 接口返回值优先映射到统一类型
- 组件 props 不直接使用 any
- WebSocket 的 payload 必须可解析、可扩展

## 8. 后端接口约定

前端开发前，后端需要至少稳定提供以下接口。

### 8.1 REST 接口

| 接口 | 方法 | 用途 |
|---|---|---|
| `/api/games` | POST | 创建游戏房间 |
| `/api/games/{gameId}/join` | POST | 加入游戏房间 |
| `/api/games/{gameId}` | GET | 获取房间与对局状态 |
| `/api/games/{gameId}/action/speak` | POST | 提交文字发言 |
| `/api/games/{gameId}/action/vote` | POST | 提交投票 |
| `/api/games/{gameId}/result` | GET | 获取结算结果 |

### 8.2 WebSocket 地址

建议统一使用：

- `ws://host/ws/game?gameId=xxx&playerId=yyy`

### 8.3 WebSocket 消息类型

| 类型 | 含义 |
|---|---|
| announce | 系统公告 |
| game_status | 游戏阶段更新 |
| role_info | 身份下发 |
| ai_speak | AI 发言 |
| vote_result | 投票结果 |
| player_update | 玩家状态变化 |
| game_over | 游戏结束 |

### 8.4 消息处理原则

- 前端收到消息后先校验 type
- 复杂消息统一交给 Store 处理
- 不在页面组件中直接写大量分支逻辑
- 断线重连与异常提示要做成通用能力

## 9. 状态管理方案

### 9.1 Store 划分

建议至少保留一个游戏主 Store，用于保存以下状态：

- gameId
- gameStatus
- myRole
- myId
- players
- chatList
- voteList
- announceList

### 9.2 必要动作

- initGame：初始化房间与对局数据
- setMyRole：设置自己身份
- setGameStatus：切换游戏阶段
- addChatMessage：追加发言
- addVote：追加投票
- addAnnounce：追加公告
- updatePlayerStatus：更新玩家存活状态
- resetGame：重置当前对局

### 9.3 状态管理要求

- 所有组件共享同一份对局数据
- 聊天记录与公告记录要可追踪
- 页面切换后状态不应丢失，除非显式重置

## 10. 核心流程设计

### 10.1 创建房间

1. 用户进入首页。
2. 点击创建游戏。
3. 前端请求后端创建房间。
4. 返回 gameId、playerId 和基础成员信息。
5. 进入房间等待页。

### 10.2 加入房间

1. 用户输入房间号。
2. 前端调用加入接口。
3. 成功后初始化 Store。
4. 建立 WebSocket 连接。
5. 进入房间等待页或直接进入对局页，取决于房间状态。

### 10.3 对局阶段流转

1. waiting：等待开局。
2. night：夜间行动阶段。
3. day：白天公告阶段。
4. speak：玩家和 AI 轮流发言。
5. vote：投票阶段。
6. end：对局结束并展示结算。

### 10.4 结束条件

- 狼人阵营人数达到胜利条件
- 好人阵营完成胜利条件
- 后端发出 game_over 事件

## 11. WebSocket 处理建议

### 11.1 连接策略

- 进入房间后建立连接
- 切换房间前主动关闭旧连接
- 页面销毁时释放连接
- 断线后进行有限次数重连

### 11.2 错误策略

- 连接失败时给出可见提示
- JSON 解析异常直接丢弃并记录日志
- 接收到未知消息类型时不阻塞主流程

## 12. 前端业务规则

### 12.1 输入规则

- 发言长度要受限
- 投票只能在合法阶段执行
- 已死亡玩家不能参与有效投票和发言
- 自己的身份只对自己可见

### 12.2 展示规则

- 存活状态应在玩家列表中实时体现
- 轮次切换应有明显公告
- 角色卡只展示当前用户自己的身份
- 结算页应展示阵营与胜负结果

## 13. 开发顺序建议

### 13.1 第一阶段：项目骨架

- 初始化 Vue3 + Vite + TypeScript
- 配置路由、Pinia、Element Plus
- 配置 ESLint、Prettier
- 建立目录结构和环境变量

### 13.2 第二阶段：基础页面

- 完成首页
- 完成房间页
- 完成对局页框架
- 完成结算页框架

### 13.3 第三阶段：状态与通信

- 完成类型定义
- 完成 Store
- 完成 REST API 封装
- 完成 WebSocket Hook

### 13.4 第四阶段：对局流程

- 打通创建/加入流程
- 打通发言与投票
- 打通阶段切换
- 打通结算展示

### 13.5 第五阶段：体验优化

- 提示与异常处理
- 页面响应式适配
- 空状态与加载态
- 日志和调试支持

## 14. 验收标准

满足以下条件即可视为前端 MVP 完成：

- 能创建和加入房间
- 能稳定连接 WebSocket
- 能显示实时公告与发言
- 能进行投票并看到结果
- 能展示角色与对局结算
- 页面结构清晰，后续扩展不需要大改

## 15. 风险与约束

### 15.1 风险

- 后端消息格式不稳定
- WebSocket 断线重连不完善
- 房间状态同步延迟
- AI 发言节奏与前端展示不同步

### 15.2 约束

- 前端不承担裁决逻辑
- 所有关键结果以后端为准
- 所有状态变更都要能追溯
